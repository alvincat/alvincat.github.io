# 1索引和散列
基本的索引类型有：顺序索引和散列索引。
顺序索引：顺序索引是基于值的顺序排序。
散列索引(hash index)：哈希索引是根据哈希函数计算关键字所属的散列桶的一种索引方法。

搜索码：用于在文件中查找记录的属性或属性集称为搜索码(search key)。

## 1.1顺序索引

顺序索引按顺序存储搜索码的值，并将搜索码与包含该搜索码的记录关联起来。

索引文件中的记录自身可以按照某种排序顺序存储。一个文件可以有多个索引，分别对应不同的搜索码。

如果包含记录的文件按照某个搜索码指定的顺序排序，那么该搜索码对应的索引称为**聚集索引**(clustering index)。聚集索引也称为**主索引**(primary key)。

如果搜索码指定的顺序与文件中记录的物理顺序不同的索引称为非聚集索引(nonclustering index)或辅助索引(secondary index)。

## 1.2稠密索引和稀疏索引
### 1.2.1 稠密索引和稀疏索引
索引记录(index record)或索引项(index entry)由一个搜索码值和指向具有该搜索码值的一个或者多个记录的指针构成。

**稠密索引**：文件中的每一个搜索码值有一个索引记录。
**稀疏索引**：只为搜索码的某些值建立索引记录。

## MySQL 主要引擎的区别
### MyISAM和InnoDB的区别
1. MyISAM是非事务安全型的，而InnoDB是事务安全型的；
2. MyISAM锁的粒度是表级，而InnoDB支持行级锁；
3. MyISAM支持全文类型索引，而InnoDB不支持全文索引；
4. MyISAM相对简单，因此在效率上优于InnoDB，小型应用可以考虑使用MyIASM。
5. MyISAM表是以文件的形式保存。在跨平台的数据转移中使用MyISAM存储会省去不少麻烦。
6. InnoDB表比MyISAM表更安全，可以在保证数据不会丢失的情况下，切换非事务表到事务表。

### MyISAM和InnoDB的应用
+ MyISAM管理非事务表。它提供高速存储和检索以及全文搜索能力。如果应用中需要执行大量的select查询，那么MyISAM是更好的选择。
+ InnoDB用于事务处理应用程序，具有众多特性，包括ACID事务支持。如果应用中需要执行大量的insert或update操作，则应该使用InnoDB，这样可以提高多用户并发操作的性能。

## 数据库范式

+ 1NF：符合1NF关系中的每个属性都不可再分
+ 2NF：在1NF基础上，消除了非主属性对于主码的部分函数依赖
+ 3NF：在2NF的基础之上，消除了非主属性对主码的传递函数依赖。
+ 消除传递依赖
+ BCNF：在 3NF 的基础上消除**主属性**对于主码的部分与传递函数依赖
+ 4NF：要求把同一个表中多对多关系删除
+ 5NF：从最终结构重新建立原始结构

### 范式中的几个依赖
+ 函数依赖：若在一张表中，在属性（或属性组）X的值确定的情况下，必定能确定属性Y的值，那么就可以说Y函数依赖于X，写作 X → Y。
+ 部分函数依赖：假如 Y 函数依赖于 X，但同时 Y 并不完全函数依赖于 X，那么我们就称 Y 部分函数依赖于 X，记作 X  P→ Y。
+ 完全函数依赖：在一张表中，若 X → Y，且对于 X 的任何一个真子集（假如属性组 X 包含超过一个属性的话），X ' → Y 不成立，那么我们称 Y 对于 X **完全函数依赖**，记作 X F→ Y。
+ 传递函数依赖：假如 Z 函数依赖于 Y，且 Y 函数依赖于 X （『Y 不包含于 X，且 X 不函数依赖于 Y』这个前提），那么我们就称 Z 传递函数依赖于 X ，记作 X T→ Z，

## 数据库中各种连接区别
连接的类型：内连接(inner join)，外连接(outer join)和交叉连接(cross join)
### 内连接
+ 等值连接：在连接条件中使用等于号(=)运算符，其查询结果中列出被连接表中的所有列，包括其中的重复列。
+ 不等连接：在连接条件中使用除等于号之外运算符
### 外连接

外连接分为左外连接，右外连接和全连接。

#### 左外连接

左外连接：返回**左表中的所有行**，如果左表中行在右表中没有匹配行，则结果中右表中的列返回空值

#### 右外连接

右外连接：返回**右表中的所有行**，如果右表中行在左表中没有匹配行，则结果中左表中的列返回空值。

#### 全外连接
全外连接：返回**左表和右表中的所有行**。当某行在另一表中没有匹配行，则另一表中的列返回空值。

### 交叉连接(笛卡尔积)
交叉连接：没有where条件子句，它将会返回被连接的两个表的笛卡尔积，返回结果的行数等于两个表行数的乘积。有where子句，往往会先生成两个表行数乘积的数据表，然后才根据where条件从中选择。

##事务
### 事务的特点(ACID)
1. 原子性(Atomicity)：原子性是指事务包含的所有操作要么全部成功，要么全部失败然后回滚。
2. 一致性(Consistency)：一致性是指事务必须使用数据库从一个一致性状态变换到另一个一致性状态，也就是说一个事务执行之前和执行之后都必须处于一致性状态。
3. 隔离性(Isolation)：多个事务并发执行时，一个事务的执行不应影响其他事务的执行。
4. 持久性(Durability)：一个事务一旦提交，它对数据库的修改应该永久保存在数据库中。即使数据库因故障而受到破坏，数据库也能够恢复(日志)。
### 事务的隔离级别

1. 未提交读(read uncommitted)：在该隔离级别中，所有事务都可以看到其他未提交事务的结果。读取未提交的数据会导致**脏读**。
2. 已提交读(read committed)：一个事务只能看到已经提交的事务所做的改变。被读取的数据可以被其他事务修改，因此可能导致**不可重复读** 。
3. 可重复读(repeatable read): 该隔离级别是**MySQL默认的隔离级别** 。它确保同一事务的多个实例在并发读取数据时，会看到同样的数据行。但该级别可能导致**幻读** 。
4. 串行化(serializable): 序列化是最高的隔离级别，他通过强制事务排序，使之不可能冲突，从而解决幻读问题。他是在每个读的数据行上加共享锁。在这个级别上，可能导致大量的超时现象和锁竞争。

### 4种隔离级别功能对比

| 隔离级别 | 脏读   | 不可重复读 | 不可重复读 |
| ---- | ---- | ----- | ----- |
| 未提交读 | 可能   | 可能    | 可能    |
| 已提交读 | 不可能  | 可能    | 可能    |
| 可重复读 | 不可能  | 不可能   | 可能    |
| 串行化  | 不可能· | 不可能   | 不可能   |


### 脏读，不可重复读，幻读

+ 脏读：脏读是指在一个事务处理过程中读取了另一个未提交的事务的数据。例如事务T1更新了一行记录的内容，但是并没有提交所做的修改。事务T2读取了修改的数据，然后T1执行了回滚操作，取消了刚才修改操作。
+ 不可重复读：在一个事务的两次查询之中数据不一致。原因可能是两次查询过程中一个新的事务更新了原有的数据。
+ 幻读：在一个事务的两次查询中数据量的不一致。例如，一个事务T1通过一个查询语句选择了几行数据，而另外一个事务T2此后插入了几行数据，导致T1两次查询的结果不一致。
## 索引

### 索引的优点
1. 可以大大加快 数据的检索速度，这也是创建索引的最主要的原因。
2. 通过创建唯一性索引，可以保证数据库表中每一行数据的唯一性。 
3. 可以加速表和表之间的连接，特别是在实现数据的参考完整性方面特别有意义。 
4. 在使用分组和排序 子句进行数据检索时，同样可以显著减少查询中分组和排序的时间。 
5. 通过使用索引，可以在查询的过程中，使用优化隐藏器，提高系统的性能。 
### 索引的缺点
1. 创建索引和维护索引要耗费时间，这种时间随着数据量的增加而增加。
2. 索引需要占物理空间，除了数据表占数据空间之外，每一个索引还要占一定的物理空间，如果要建立聚簇索引，那么需要的空间就会更大。
3. 当对表中的数据进行增加、删除和修改的时候，索引也要动态的维护，这样就降低了数据的维护速度。


### 需要创建索引的情形
+ 在经常需要搜索的列上，可以加快搜索的速度；
+ 在作为主键的列上，需要强制该列的唯一性和组织表中数据的排列结构；
+ 在**经常用在连接的列上**，这些列主要是一些外键，可以加快连接的速度；
+ 在经常需要**根据范围进行搜索的列上**创建索引。因为索引已经排序，其指定的范围是连续的；
+ 在经常需要排序的列上创建索引。因为索引已经排序，这样查询可以利用索引的排序，加快排序查询时间；
+ 在经常使用在WHERE子句中的列上面创建索引，加快条件的判断速度。


### 不应创建索引的情形
+ 对于在查询中很少使用或者参考的列不应该创建索引。这是因为，既然这些列很少使用到，因此有索引或者无索引，并不能提高查询速度。相反，由于增加了索引，反而降低了系统的维护速度和增大了空间需求。
+ 只有很少数据值的列也不应该增加索引。这是因为，由于这些列的取值很少，例如人事表的性别列，在查询的结果中，结果集的数据行占了表中数据行的很大比例，即需要在表中搜索的数据行的比例很大。增加索引，并不能明显加快检索速度。
+ 定义为text, image和bit数据类型的列不应该增加索引。这是因为，这些列的数据量要么相当大，要么取值很少。
+ 当修改性能远远大于检索性能时，不应该创建索引。这是因为，**修改性能和检索性能是互相矛盾的**。当增加索引时，会提高检索性能，但是会降低修改性能。当减少索引时，会提高修改性能，降低检索性能。因此，当修改性能远远大于检索性能时，不应该创建索引。


### 索引类别
1. 唯一索引：不允许其中任何两行具有相同索引值的索引。
2. 主键索引：主键索引是唯一索引的特定类型。该索引要求主键中的每个值都唯一。
3. 聚集索引：表中行的物理顺序与键值的逻辑（索引）顺序相同。一个表只能包含一个聚集索引。聚集索引相对于非聚集索引具有更快的数据访问速度。
4. 非聚集索引：表中行的物理顺序与键值的逻辑顺序不匹配。


### B-树和$B^+树$
#### B-树

B树的属性：
1. B树是一种多路平衡搜索树，定义任意非叶子结点最多有M个孩子且M>2;
2. 根结点的儿子个数为[2, M];
3. 除根节点以外的非叶子结点的孩子个数为[M/2, M]；
4. 每个结点最少存放$\lceil M/2 -1 \rceil$ 和至多M-1个关键字；
5. 非叶子结点关键字个数 = 指向孩子结点的指针个数 - 1；
6. 非叶子结点的关键字k[1],k[2],...,k[M-1]满足k[i] < k[i+1]；
7. 非叶子结点的指针p[1],p[2],...,p[M]满足条件：p[1]指向关键字小于k[1]的子树，而p[i]指向关键字在$(k[i-1], k[i])$范围(开区间)内的的子树。
8. 所有叶子结点位于同一层。

B树的特性：
1. 关键字集合分布在整颗树中；
2. 任何一个关键字出现且只出现在一个结点中；
3. 搜索有可能在非叶子结点结束；
4. 其搜索性能等价于在关键字集合内做二分查找；
5. 具有自动层次控制能力

### $B^+树$

$B^+$ 树是B树的变体，也是多叉平衡搜索树。但又有一些不同之处：
1. 非叶子结点的子树指针与关键字个数相同；
2. 非叶子结点的子树指针p[i]指向关键字值属于$[k[i], k[i+1]]$ 范围(闭区间)的子树；
3. 为所有叶子结点增加一个链接指针；
4. 所有关键字都在叶子结点出现；

$B^+$树的特性：
+ 所有关键字都出现在叶子结点的链表中(稠密索引)，且链表中的关键字是有序的；
+ 不可能在非叶子结点命中；
+ 非叶子结点相当于叶子结点的索引(稀疏索引)，叶子结点相当于存储关键字的数据层；
+ 更适合作为文件索引系统。
